# Prompt for Claude Code — Add RoadScope Live Support to MeasurePRO (Electron)

> **Origin:** Generated by Replit AI from the RoadScope backend implementation.
> **Status:** Not yet implemented. Save for when the RoadScope live-support backend is confirmed working.
> **Prerequisites:** RoadScope `/api/live-support/*` endpoints + WebSocket handler must be live and tested before starting MeasurePRO side.

---

## Goal

Add a "Live Support" feature to the MeasurePRO Electron app so that any authenticated MeasurePRO user can request live screen-share + annotation help from a RoadScope admin. The admin side already exists in the RoadScope web app and must not be re-implemented. Your job is to build the MeasurePRO end-user side and make it talk to the existing RoadScope live-support backend (REST + WebSocket + Firestore audit) using the exact same contract documented below.

When done, a RoadScope admin opening the existing admin Live Support page must be able to enter a session code generated inside MeasurePRO, request to join, get approved by the MeasurePRO user, and run a full session (screen share, annotations, direction swap, end) without any backend changes.

## Non-goals

- Do not rebuild the admin UI, the admin REST endpoints, the audit history page, or the Firestore schema.
- Do not change the RoadScope server. MeasurePRO must conform to the existing API and WebSocket contract.
- Do not invent new message types or fields. Match the contract byte-for-byte.

## Architecture

- The RoadScope backend is the single source of truth for sessions. MeasurePRO is just another authenticated client.
- Auth uses Firebase (same Firebase project as RoadScope). The Electron app must sign the user in with Firebase and attach a fresh Firebase ID token to every REST and WebSocket call.
- REST base URL is configurable via env (e.g. `ROADSCOPE_API_BASE_URL`, default `https://app.roadscope.example`). All live-support routes are under `/api/live-support/*`.
- WebSocket signaling URL is `${WS_BASE}/api/live-support/ws` where `WS_BASE` is the same host using `wss://` for https and `ws://` for http. Pass `session`, `role`, `token` as query params.
- Real-time media uses WebRTC (`RTCPeerConnection`) directly between admin and user. Annotations and cursor updates use an `RTCDataChannel` named `annotations`, with WebSocket fallback through the same signaling channel.

## Auth contract

- Every REST call must send `Authorization: Bearer <Firebase ID token>` from `firebase/auth` `getIdToken()` (force refresh when needed).
- Every WebSocket connection must include the same token in the `token` query param.
- The MeasurePRO user is always `role=user` over WebSocket. Never connect as `admin` from MeasurePRO.

## REST contract (use exactly these endpoints, methods, and bodies)

### Create a session (user)

```
POST /api/live-support/sessions
Auth: user.
Body: none.
Response: { ok: true, session: { id, state: 'pending', userId, userEmail, createdAt, direction: 'user-to-admin', directionsUsed: [] } }
```

Display `session.id` to the MeasurePRO user as the code they read out to support.

### Poll session status (user or admin)

```
GET /api/live-support/sessions/:id
Auth: user (must be session owner) or admin.
Response: { ok: true, session: <sanitized> }
```

### Approve admin request (user)

```
POST /api/live-support/sessions/:id/approve
Allowed only when state === 'awaiting-approval'.
```

### Deny admin request (user)

```
POST /api/live-support/sessions/:id/deny
Allowed only when state === 'awaiting-approval'. Ends the session.
```

### End session (either side)

```
POST /api/live-support/sessions/:id/end
```

### Swap presenter direction (either side, while active)

```
POST /api/live-support/sessions/:id/swap
Toggles direction between 'user-to-admin' and 'admin-to-user'.
```

You do **not** call `/api/live-support/admin/request` or `/api/live-support/admin/history` from MeasurePRO. Those are admin-only.

### Sanitized session object shape used everywhere:

```json
{
  "id": "string",
  "state": "pending | awaiting-approval | active | ended | expired",
  "userId": "string",
  "userEmail": "string",
  "adminId": "string?",
  "adminEmail": "string?",
  "createdAt": "ISO string",
  "approvedAt": "ISO string?",
  "endedAt": "ISO string?",
  "direction": "user-to-admin | admin-to-user",
  "directionsUsed": ["string"],
  "endedBy": "user | admin | system?",
  "endReason": "string?"
}
```

### Server-side TTLs you must accommodate in UI:

- Pending / awaiting-approval auto-expire after **5 min**.
- Active sessions auto-end after **30 min** of inactivity.
- Ended/expired sessions are purged from memory after **10 min** (audit stays in Firestore).

## WebSocket signaling contract

Connect to: `${WS_BASE}/api/live-support/ws?session=<SESSION_ID>&role=user&token=<idToken>`

**Inbound messages from server:**

| Type | When | Payload |
|------|------|---------|
| `connected` | On connect | `{ session, peerConnected }` |
| `peer-connected` | Admin connects | `{}` |
| `peer-disconnected` | Admin disconnects | `{ role }` |
| `admin-request` | Admin wants to join | `{ adminEmail, session }` — show approve/deny UI |
| `session-approved` | User approved | `{ direction, session }` — to both peers |
| `direction-changed` | Direction swapped | `{ direction, session }` — tear down PC, re-offer if presenter |
| `session-ended` | Session ended | `{ endedBy, endReason, session }` — clean up, auto-close after ~4s |

All relayed peer messages arrive verbatim with `from: 'user' | 'admin'` added.

**Outbound messages MeasurePRO sends:**

```
WebRTC: { type: 'offer', sdp }, { type: 'answer', sdp }, { type: 'ice-candidate', candidate }
Capture lifecycle: { type: 'ready' }, { type: 'bye' }
Annotation overlay (also valid over the data channel):
  { type: 'cursor', x, y }                    — normalized 0..1, throttle ~30 ms
  { type: 'annotation', tool, color, points, id, createdAt? }
  { type: 'annotation-clear' }
```

Annotations fade after **5000 ms** (`ANNOTATION_TTL_MS`).

## WebRTC flow MeasurePRO must implement

1. Open the WebSocket as `role=user` as soon as a `sessionId` exists.
2. Build an `RTCPeerConnection` with ICE servers `stun:stun.l.google.com:19302` and `stun:stun1.l.google.com:19302`.
3. Wire `pc.onicecandidate` → send `{ type: 'ice-candidate', candidate }` over WS.
4. Wire `pc.ontrack` → set `remoteStream` and play it in a `<video>` element.
5. Wire `pc.ondatachannel` → accept incoming data channel, parse JSON messages, route to annotation handler.
6. **Presenter decision:** `iAmPresenter = session.direction === 'user-to-admin'`. When the session becomes active and `iAmPresenter`:
   - Start screen capture (see Electron section).
   - Add tracks (avoid duplicate senders).
   - Create an `RTCDataChannel('annotations')` on the first offer.
   - Create offer and send `{ type: 'offer', sdp }`.
7. Handle inbound:
   - `offer` → `setRemoteDescription`, `createAnswer`, send `{ type: 'answer', sdp }`.
   - `answer` → `setRemoteDescription`.
   - `ice-candidate` → `addIceCandidate`.
   - `cursor | annotation | annotation-clear` → forward to annotation handler.
   - `direction-changed` → tear down PC, clear remote stream; if you become presenter, kick off `createOffer` again.
   - `session-ended` → tear down PC, stop capture, clear remote stream.
8. `sendData(msg)` should prefer the data channel when open and fall back to the WebSocket.

## Electron screen capture

- In `main.ts`, register `session.defaultSession.setDisplayMediaRequestHandler` (Electron 27+) and present a source picker (e.g. small `BrowserWindow` listing `desktopCapturer.getSources({ types: ['screen', 'window'] })`). Resolve with the chosen `{ video: source, audio: 'loopback' | undefined }`.
- In the renderer, call `navigator.mediaDevices.getDisplayMedia(...)` exactly as the web app does. Do not use the legacy `chromeMediaSource: 'desktop'` constraints unless targeting older Electron — prefer the official handler API for Electron >= 27.
- Stop tracks and send `{ type: 'bye' }` when the source ends or the user clicks End.

## UI requirements (MeasurePRO renderer)

- A **"Live Support"** entry point in the MeasurePRO chrome (e.g. Help menu and a button somewhere persistent).
- **No active session:** modal "Start session" → calls `createSession()` → displays the session code prominently (large, monospace, copy button) plus instructions to read it to support.
- **`awaiting-approval`:** show "Admin \<email\> wants to join" with **Approve** and **Deny** buttons.
- **`active` as presenter** (`direction === 'user-to-admin'`): show local preview thumbnail; full screen is being shared. Show a Stop Sharing / End button and a Swap Direction button.
- **`active` as viewer** (`direction === 'admin-to-user'`): full-size `<video>` with admin's stream and an `AnnotationCanvas` overlay.
- **Floating bar** (bottom-right) must remain when the modal is closed so users never lose track of an in-progress session. Show: state, agent connected indicator, direction label, session id, Open and End actions. The session and PC must keep running while a sessionId exists — do not unmount on route change.
- **On session-ended**, show a brief reason then auto-close after 4s.
- **Annotation canvas**: cursor (ephemeral, throttled ~30 ms), arrow (persisted shape), freehand (persisted line); fade after 5000 ms; Clear button sends `{ type: 'annotation-clear' }`.

## Errors and edge cases

- **401/403 from REST** → refresh Firebase token once and retry; otherwise show "Please sign in again".
- **409** from `admin/request` is admin-side only; treat any non-2xx from REST as a user-visible failure with retry.
- **WebSocket close while active** → attempt one reconnect with same query params; if that fails, surface "Connection lost" with manual retry.
- **`peer-disconnected`** → keep session alive, show "Agent disconnected, waiting…".
- **Multiple displays / multi-monitor**: source picker must let user pick. Persist nothing — re-prompt every session.
- **macOS screen-recording permission**: detect `systemPreferences.getMediaAccessStatus('screen')`; if not granted, show instructional dialog.
- **Windows**: ensure the Electron window is not captured into its own share by default (offer "Share entire screen" vs "Share a window").

## Acceptance criteria

1. Signed-in MeasurePRO user can press Live Support, get a session code, and read it to a RoadScope admin.
2. RoadScope admin (existing UI) can join, MeasurePRO user sees Approve/Deny prompt, on Approve admin sees MeasurePRO screen via WebRTC.
3. Admin can draw cursor/arrow/freehand annotations and clear them; MeasurePRO user sees them with same TTL fade.
4. Either side can swap direction; the other side renegotiates and the new presenter's stream appears.
5. Either side can end; both sides clean up media and PC; audit log entry in Firestore reflects session correctly.
6. Closing the modal during active session leaves a floating bar; reopening returns to live session without restarting.
7. All TTLs (5 min pending, 30 min inactivity, 10 min purge, 5s annotation fade, 30 ms cursor throttle) are honored.
8. **No changes required to the RoadScope server, admin UI, or Firestore schema.**

## Reference contract (do not deviate)

- REST base path: `/api/live-support`
- WS path: `/api/live-support/ws` with `?session=<id>&role=user&token=<idToken>`
- Firestore audit collection: `liveSupportSessions` (read-only from MeasurePRO)
- WS send types: `offer`, `answer`, `ice-candidate`, `ready`, `bye`, `cursor`, `annotation`, `annotation-clear`
- WS receive types: `connected`, `peer-connected`, `peer-disconnected`, `admin-request`, `session-approved`, `direction-changed`, `session-ended`, plus relayed peer messages with `from: 'admin'`
