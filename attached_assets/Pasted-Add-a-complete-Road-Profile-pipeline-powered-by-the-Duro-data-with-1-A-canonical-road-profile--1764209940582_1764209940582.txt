PART 1 – DATA MODEL: CANONICAL ROAD PROFILE
----------------------------------------------------------------------

1. Create or extend a central “road profile” model that will be derived from the Duro stream.
   - If a similar model exists, extend it; if not, create a dedicated module, e.g.:
     - src/core/roadProfile/roadProfileTypes.ts
     - src/core/roadProfile/roadProfileProcessor.ts

2. Define a strongly typed interface for a road profile POINT, something like:

   export type GradeAlertType = "NONE" | "GRADE_12_UP" | "GRADE_12_DOWN";
   export type KAlertType = "NONE" | "K_OVER_10";

   export interface RoadProfilePoint {
     profileId: string;            // link to survey/route
     chainage_m: number;           // distance from profile start, in meters (0 at start)
     lat: number;                  // WGS84
     lon: number;                  // WGS84
     elev_m: number;               // elevation (be consistent with existing unit/source)
     grade_pct: number;            // signed grade, % (up positive, down negative)
     k_factor: number;             // curvature / vertical transition metric (see below)
     speed_kmh?: number;           // optional, if available from Duro
     timestamp_iso: string;
     quality: string;              // e.g. Duro fix type / quality indicator
     grade_alert_type: GradeAlertType;
     k_alert: KAlertType;
   }

3. Chainage and sampling:
   - Use the Duro GNSS positions along the survey path to compute cumulative distance along the track (chainage).
   - Either:
     - a) compute chainage directly from each GNSS point (no resampling), or
     - b) resample to a regular interval (e.g. every 1 m, or reuse any existing sampling strategy).
   - Prefer reusing any existing distance utilities if they exist (e.g. haversine, projected distance functions).
   - The important point: each RoadProfilePoint must have a monotonically increasing chainage_m.

4. Grade calculation (grade_pct):
   - Compute grade using elevation change over horizontal distance.
   - For each point i:
     - Take a small window around i (e.g. previous and next point or an existing smoothing window) to avoid noisy spikes.
     - grade_pct = (Δelev / Δdistance) * 100, signed (positive = uphill).
   - Reuse any existing smoothing / filter utilities if already implemented.

5. K-factor calculation (k_factor):
   - Implement K-factor as a measure of how quickly grade is changing.
   - For now, implement a simple approach that can be refined later:
     - k_factor = absolute difference in grade_pct between two consecutive samples OR over a small window.
       e.g. k_factor = |grade_pct(i+1) - grade_pct(i)| (or a small symmetric window).
   - Keep this logic in a small helper function so we can later swap the definition to a strict highway-engineering K if needed.

6. Alerts:
   - For each RoadProfilePoint, compute:
     - grade_alert_type:
       - "GRADE_12_UP"   if grade_pct >= +12
       - "GRADE_12_DOWN" if grade_pct <= -12
       - "NONE" otherwise
     - k_alert:
       - "K_OVER_10" if k_factor > 10
       - "NONE" otherwise
   - Thresholds should be declared as constants/config in a single place, e.g. src/core/roadProfile/constants.ts:
     - GRADE_UP_ALERT_THRESHOLD = 12;
     - GRADE_DOWN_ALERT_THRESHOLD = -12;
     - K_FACTOR_ALERT_THRESHOLD = 10;

7. Integration:
   - Add a main function, e.g.:

     export function buildRoadProfileFromDuroStream(params: {
       surveyId: string;
       rawDuroSamples: DuroSample[]; // reuse existing Duro sample type
     }): RoadProfilePoint[] { ... }

   - This function should:
     - sort by time,
     - compute chainage,
     - compute grade_pct, k_factor,
     - set alerts,
     - return the RoadProfilePoint[].

   - Wire this into the part of the pipeline where survey data are finalized (e.g. when user ends a run or when exporting).

PART 2 – ALERT SEGMENTS / CRITICAL LOCATIONS
----------------------------------------------------------------------

1. On top of the per-point alerts, build summarized ALERT SEGMENTS:
   - Define an interface:

     export type AlertType = "GRADE_12_UP" | "GRADE_12_DOWN" | "K_OVER_10";

     export interface RoadProfileAlertSegment {
       alert_id: string;
       alert_type: AlertType;
       from_chainage_m: number;
       to_chainage_m: number;
       length_m: number;
       max_grade_pct: number;
       min_grade_pct: number;
       max_k_factor: number;
       lat: number;   // representative location (e.g. midpoint)
       lon: number;
       notes?: string;
     }

2. Build segments:
   - Scan the RoadProfilePoint[] and group contiguous points where:
     - grade_alert_type is GRADE_12_UP or GRADE_12_DOWN, or
     - k_alert is K_OVER_10.
   - Each contiguous “run” of points for the same alert type should become one RoadProfileAlertSegment.
   - Compute:
     - from_chainage_m = first point’s chainage
     - to_chainage_m = last point’s chainage
     - length_m = to_chainage_m - from_chainage_m
     - max_grade_pct, min_grade_pct across the segment
     - max_k_factor across the segment
     - lat/lon = location of the point with max absolute grade or midpoint
   - notes can be left empty or filled with a simple auto-generated description string (e.g. "Steep climb (13.6% up)" or "Sharp vertical transition (K>10)").

3. Implement a helper like:

   export function computeRoadProfileAlertSegments(points: RoadProfilePoint[]): RoadProfileAlertSegment[] { ... }
