--- ROOT CAUSE ---
The trigger model is backwards. In POISelector.tsx (line 52), selecting a POI type
immediately calls triggerActionForType. KeyboardShortcutHandler does the same at
line 117. Both bypass all prerequisite checks and create phantom log entries with
no valid measurement, no active survey, and wrong note text.

For height-clearance POIs (auto-capture-and-log), the laser measurement arriving
through the measurement pipeline must be the ONLY trigger for creating a log entry.
For measurement-free POIs (auto-capture-no-measurement), selection is the trigger
— but only when all guards pass.

--- TASK 1: Remove immediate trigger for auto-capture-and-log ---
In POISelector.tsx (line 52) and KeyboardShortcutHandler.tsx (line 117):
- When a POI type with action 'auto-capture-and-log' is selected (dropdown or
  keyboard shortcut), ONLY set the active POI type in the store.
- Do NOT call triggerActionForType or handleAutoCaptureAndLog.
- Do NOT create any log entry.
- The active POI type stays set until the user explicitly changes it. Do NOT
  auto-reset to '' after a capture. A driver may run for hours logging the same
  POI type (e.g. "Wire") with zero manual interaction — the pipeline fires on
  every qualifying measurement as long as the type remains selected.

--- TASK 2: Wire handleAutoCaptureAndLog into the measurement pipeline ---
CRITICAL: handleAutoCaptureAndLog is currently ONLY called from POISelector and
KeyboardShortcutHandler. After Task 1 removes those calls, there will be NO code
path that invokes it. You must wire it into useMeasurementLogging.ts.

In useMeasurementLogging.ts, at the point where a valid measurement is processed:
- Check if the active POI type has action 'auto-capture-and-log'
- If yes, proceed based on the active detection mode:

  SINGLE-SHOT MODE:
  - If the measurement is within ignore-above / ignore-below thresholds, call
    handleAutoCaptureAndLog immediately with the single measurement.
  - If out of threshold, silently discard — no entry, no toast.

  COUNTER DETECTION & BUFFER DETECTION MODES:
  - These modes collect multiple measurements for one physical object before
    committing. Each individual measurement is added to a temporary set.
  - When the detection cycle completes (counter reached or buffer window closes),
    pass the FULL set of collected measurements to handleAutoCaptureAndLog.
  - Do NOT call handleAutoCaptureAndLog on each individual measurement — only
    once per completed detection cycle, with all readings in the set.
  - Out-of-threshold measurements are excluded from the set silently.

This wiring is the most critical step. Without it, height-clearance POI logging
is completely broken after Task 1.

--- TASK 5: handleAutoCaptureAndLog — measurement handling and note format ---
The function receives either a single measurement (single-shot) or an array of
measurements (counter/buffer detection).

SINGLE-SHOT:
  - rel (height column): the single measurement value
  - note: "Height: X.XXm | GND REF: X.XXm | {label}"

COUNTER / BUFFER DETECTION (array of measurements passed in):
  - rel (height column): the LOWEST value in the measurement set
  - note: "Readings: X.XXm, X.XXm, X.XXm | Min: X.XXm | GND REF: X.XXm | {label}"
    (list all raw readings in the set, then the minimum, then GND REF, then label)

RULES THAT APPLY TO BOTH:
  - GND REF must ALWAYS appear in the note, even when the value is 0.00m.
    It is required for audit and traceability — never omit it.
  - Remove all "Visual documentation" text from notes entirely.
  - The label comes from the active POI type's display name.
