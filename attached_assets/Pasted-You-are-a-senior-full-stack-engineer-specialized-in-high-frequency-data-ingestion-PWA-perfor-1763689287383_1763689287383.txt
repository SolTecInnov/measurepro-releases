TASK 1 – MULTI-WORKER ARCHITECTURE (MANDATORY)

Implement a clear multi-worker architecture. NO worker is allowed to write directly to IndexedDB. Only a single “orchestrator” is responsible for writes and global consistency.

Create at least the following Web Workers:
	1.	gpsWorker

	•	Reads GPS, validates, normalizes (lat/lng/alt/speed/accuracy).
	•	Attaches precise timestamps (monotonic if available).
	•	Sends small, lightweight GPS updates to orchestratorWorker.

	2.	laserWorker

	•	Reads LiDAR distance values at high frequency.
	•	Detects relevant “events” (e.g. when current measurement is within configured thresholds).
	•	Attaches timestamps to all events.
	•	Receives the current “active POI type” from the main thread/orchestrator (set when user presses a remote button: wire, branch, overpass, etc.).
	•	For each detected event, sends a compact message to orchestratorWorker with: timestamp, raw distance, derived height info, current active POI type.

	3.	photoWorker

	•	Manages a rolling buffer of recent frames coming from the camera.
	•	Buffer length: about 1–2 seconds of history.
	•	Each frame has a timestamp.
	•	Frames must be compressed (e.g. WebP, medium quality) and resized (e.g. 640x360 or 800x450) to stay lightweight.
	•	On request from orchestratorWorker, returns the frame whose timestamp best matches “eventTimestamp – configuredOffset” (e.g. 1s).
	•	Ensures that even under UI lag, we always retrieve the correct frame corresponding to the LiDAR event.

	4.	timelapseWorker

	•	Builds a timelapse as a single video OR efficient sequence (NOT thousands of individual JPEGs).
	•	Receives frames or frame references and outputs a compact format (or at least a structured sequence easy to export as a single file).
	•	Maintains a mapping between “survey time” and timelapse time/index, so POIs can reference a specific frame/time in the timelapse.

	5.	videoWorker (optional but preferred)

	•	Handles continuous video stream if enabled.
	•	Maintains a mapping between POI event timestamps and video timeline (time offsets) for later playback.

	6.	mapWorker

	•	Responsible for map features generation: POI markers, geometry, clustering preparation.
	•	Receives event summaries from orchestratorWorker and precomputes data for map rendering, so the main thread only has to render.

	7.	orchestratorWorker (THE BOSS)

	•	Receives messages from all other workers.
	•	Maintains in-memory queues and lookup tables keyed by timestamp or event id.
	•	When a LiDAR event arrives:
	•	Resolves the active POI type.
	•	Finds the closest GPS sample (or interpolates if needed).
	•	Requests the correct image frame from photoWorker based on timestamp.
	•	Associates timelapse/video reference (frame index or time).
	•	Builds a single, atomic “POI event record” object that contains:
	•	id
	•	timestamp
	•	POI type
	•	LiDAR measurement and metadata
	•	GPS snapshot
	•	image blob or reference
	•	timelapse/video pointer
	•	any extra metadata (e.g. survey id, vehicle id, etc.)
	•	Handles batching and writing to IndexedDB (see Task 2).
	•	Publishes condensed status updates to the main thread for UI (queue size, write rate, errors, etc.).
	•	Enforces strict ordering and avoids concurrent transactions.

TASK 2 – INDEXEDDB WRITING STRATEGY (ZERO-LAG)

Redesign IndexedDB usage for performance and safety.
	1.	Single writer model

	•	Only orchestratorWorker opens IndexedDB and writes data.
	•	NO other worker or UI code touches the DB directly.

	2.	Batching

	•	Maintain an in-memory queue of POI records.
	•	Batch size target: around 20–50 items per transaction (configurable).
	•	Flush conditions:
	•	A timer (e.g. every 500–800ms).
	•	OR queue size exceeds batch size.
	•	Use a single readwrite transaction per batch.
	•	Avoid opening multiple overlapping transactions.

	3.	Memory guardrail

	•	Set a hard cap on the in-memory queue size (e.g. 100–150 events).
	•	If queue reaches the cap:
	•	Temporarily throttle intake from workers (e.g. backpressure messages).
	•	Prioritize flushing data to IndexedDB.
	•	Expose current queue size and backpressure state to UI (for /debug).

	4.	Crash-tolerant batching

	•	Each “POI event record” must be stored as a single object in IndexedDB, not fragmented across multiple stores.
	•	Use a clear schema, for example:
	•	store: “poiEvents”
	•	keyPath: “id”
	•	indexes: by timestamp, by surveyId, by poiType, etc.
	•	If the app crashes, all fully committed batches are intact. Only the current in-memory queue might be lost.

TASK 3 – PERFECT SYNCHRONIZATION LOGIC

Implement deterministic synchronization based on timestamps:
	•	When the user presses a POI button (remote or UI), update “currentActivePoiType” with a timestamp.
	•	When laserWorker flags an event:
	•	Ensure event captures the POI type that was active at that moment.
	•	For each LiDAR event, orchestratorWorker must:
	•	Match to nearest GPS sample in time (and optionally interpolate).
	•	Request from photoWorker: frame closest to (eventTimestamp – configuredOffset).
	•	Associate timelapse/video references based on that timestamp.
	•	Each stored POI event must contain:
	•	poiId
	•	surveyId
	•	createdAt (timestamp)
	•	poiType
	•	lidar: rawDistance, derivedClearance, etc.
	•	gps: lat, lng, alt, speed, accuracy, etc.
	•	image: compressed data or reference key
	•	timelapseRef: time or frame index
	•	videoRef: time or frame index (if applicable)

The architecture must guarantee:
	•	No event is stored without GPS,
	•	No event is stored without a resolved POI type,
	•	No event refers to a “null” image unless explicitly allowed by config.
